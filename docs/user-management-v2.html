User Management v2.0 — Backend Showcase

How the Kalnostics New backend models staff users: one identity, one employment record per business, and a different role at every branch. Registration, branch assignment, and permission resolution — end to end.

NestJS · Prisma · PostgreSQL (RLS) · /api/v1/users/manage

1 The data model

Three tiers: a global Person, one TenantStaffMembership per business, and one UserBranchProfile per branch assignment (the user↔branch↔role↔module junction). Permissions and module enablement hang off the branch.

One role per branch is enforced by a partial unique index ubp_person_branch_active_unique on (tenant_id, person_id, branch_id) where the row is active. The role can still differ from branch to branch.

2 One user, different roles per branch

The role lives on each branch assignment (profileKey), not on the membership. The membership's roleKey is only an optional primary/default and is never forced onto the branches.

3 Register first, assign roles later

A user can be created with just basic details (no role). Roles + modules are assigned per branch afterwards, as an array of objects.

4 How permissions resolve (per branch)

The baseline comes from the role at that branch; stored overrides win; only modules the branch runs are shown. 

flowchart TD
  S([GET .../branch-permissions?branchId]):::start --> A[Find the user's UserBranchProfile
for this branch -> its role]:::step
  A --> Z{Active assignment?}:::choice
  Z -- No --> R0[Return empty list]:::stop
  Z -- Yes --> B[Read modules enabled for the branch
BranchModule.isEnabled]:::step
  B --> C["baseline = ROLE_TEMPLATES[role].permissions"]:::step
  C --> D[Load UserBranchPermission overrides
allow / deny]:::step
  D --> E["For each catalogue permission in an enabled module:
allowed = override ?? baseline"]:::step
  E --> R([Return module-grouped permissions]):::start

A role template carries two independent lists — permissions and linked modules. The module link is optional; when a template links specific modules, an assignment's moduleId must be one of them.

5 Two independent on/off switches

LayerFieldEffect
Global TenantStaffMembership.status INACTIVE blocks login to the whole tenant; login is rejected outright.
Per-branch UserBranchProfile.branchStatus INACTIVE blocks only that branch; the JWT/profile switcher excludes it. Global wins if it is INACTIVE.

6 Endpoints — /api/v1/users/manage

Business-authenticated. Tenant + actor come from the JWT, never the body. Responses use the global { success, data, meta } envelope. (The pre-v2.0 /users controller was removed.)

MethodPathPurpose
POST/users/manageCreate a staff user (role optional)
GET/users/manageList users — paginated, filterable, sortable; each row includes a branches[] array with per-branch role, module & status
GET/users/manage/:idFull detail (Aadhaar masked; per-branch roles)
PATCH/users/manage/:idEdit identity / password / primary role / status
POST/users/manage/:id/photoUpload photo (JPG/PNG ≤ 2 MB)
POST/users/manage/:id/branchesAssign branches — [{ branchId, roleKey, moduleId }]
PATCH/users/manage/:id/branches/:branchIdPatch one branch (role / module / status / default)
PATCH/users/manage/:id/activate · /deactivateGlobal status
GET/users/manage/:id/branch-permissionsResolve (user+branch) permissions
PUT/users/manage/:id/branch-permissionsReplace (user+branch) overrides
GET/users/manage/roles · /modules · /permissionsCatalogues & permissions screen

8 Worked example

Asha is created with no role, then assigned two branches with different roles. Her effective permissions differ by branch because the baseline is the role at that branch.

StepCallResult
1 POST /users/manage (no role) Person + membership created · roleKey = null · USR-00042
2 POST /users/manage/:id/branches doctor @ Main Lab (default), branch_admin @ City Branch
3 GET …/branch-permissions?branchId=MainLab baseline = doctor template (registration, lab ops, opd, radiology ∩ enabled)
4 GET …/branch-permissions?branchId=City baseline = branch_admin template — different permissions, same user
5 PATCH …/branches/CityBranch { roleKey: "receptionist" } Only City Branch is re-roled; Main Lab unchanged
6 PATCH …/branches/CityBranch { branchStatus: "INACTIVE" } Asha loses access to City Branch only; still a doctor at Main Lab
