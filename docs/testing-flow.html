Complete Integration Testing Flow

This guide explains the end-to-end flow of the Kalnostics New project. Fill in the values in the sidebar as you progress, and the cURL commands below will update dynamically. Copy and run them directly in your shell.

Step 1.1

Initial Local Environment Setup

Database & Env

Ensure your environment variables are configured and the database has been initialized with the schema and the Row-Level Security (RLS) configurations.

  • Configure your database URL and secrets. Copy .env.example to .env and adjust DATABASE_URL and JWT_SECRET.
  • Install dependencies: run pnpm install in the root directory.
  • Run migrations to push schema structure to PostgreSQL: npx prisma db push or npx prisma migrate dev.
  • Apply Row-Level Security Policies: Run the RLS script manually against your PostgreSQL database:
    psql "$DATABASE_URL" -f prisma/rls.sql
Step 1.2

Seed the Initial SUPER_OWNER SiteAdmin

Database Seeding

To register new tenants (businesses) and set up the system, you must have an initial super admin account. Seed the database with the pre-configured seeder script:

Terminal Command 
npx prisma db seed

Seeded Credentials

The seeder creates a default SUPER_OWNER SiteAdmin user with these credentials:

  • Email: admin@kalnostics.com
  • Password: SuperSecret1
Step 1.3

Start the NestJS Backend Server

Runtime

Start the server in watch mode. By default, it runs on port 3000. Verify that it starts cleanly and connects to the database.

Terminal Command 
pnpm run start:dev
Step 2.1

SiteAdmin Authenticate

POST /api/v1/siteadmin/auth/login public
SiteAdmin Auth

Log in as the default SiteAdmin to receive an 8-hour SiteAdmin access JWT. This token bypasses the business logic rules and lets you manage the platform.

cURL Request 
curl -X POST {HOST}/api/v1/siteadmin/auth/login \
  -H "Content-Type: application/json" \
  -d '{
    "email": "admin@kalnostics.com",
    "password": "SuperSecret1"
  }'
Expected Response

  "success": true,
  "data": {

  },
  "meta": {
    "timestamp": "2026-06-04T11:00:00.000Z"

}

Next Step

Copy the accessToken from the response, paste it into the **SiteAdmin Token ($SA_TOKEN)** input field in the sidebar, and proceed.

Step 3.2

Change the Admin's Temporary Password

POST /api/v1/auth/change-password business JWT
Security

Temporary passwords should be changed immediately upon first login. Update the password to a secure new value (must contain at least 8 characters, 1 uppercase letter, and 1 digit).

cURL Request 
curl -X POST {HOST}/api/v1/auth/change-password \
  -H "Authorization: Bearer {BIZ_TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{
    "currentPassword": "{TEMP_PASSWORD}",
    "newPassword": "NewSecurePassword1"
  }'

Step 4.4

Configure Custom Profile Permissions Overrides

PUT /api/v1/users/:id/permissions business JWT
Permissions Override

Tenant admins can override specific baseline permissions for a staff member at a branch level (allowing or denying explicitly).

cURL Request 
curl -X PUT {HOST}/api/v1/users/{STAFF_ID}/permissions \
  -H "Authorization: Bearer {BIZ_TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{
    "branchId": "{BRANCH_ID}",
    "profileKey": "doctor",
    "overrides": [
      { "code": "billing:read", "override": "allow" },
      { "code": "report:verify", "override": "deny" }
    ]
  }'

Check Effective Permissions

Step 5.1

Understanding Row-Level Security (RLS)

Architecture Detail

Kalnostics New enforces database-level tenant isolation. During connection setup, the API runs SET LOCAL app.current_tenant_id = 'tenant-uuid'. The database tables (like branches, user_branch_profiles) have row-level policies matching that ID.

As a result, a user authenticated with Tenant A will never see Tenant B's data, even if they guess the resource UUID.

Step 5.2

Test RLS Cross-Access Block

GET /api/v1/branches/:id business JWT
Security Audit

To verify the tenant isolation boundary, create a second tenant (Tenant B) and use its Business Token to fetch the branch of Tenant A ($BRANCH_ID). The endpoint should return a 404 Not Found rather than revealing the details or throwing a generic access violation error.

cURL Request 
curl {HOST}/api/v1/branches/{BRANCH_ID} \
  -H "Authorization: Bearer paste-tenant-b-token-here"
Expected Response
{
  "success": false,
  "error": {
    "code": "BRANCH_NOT_FOUND",
    "message": "Branch {BRANCH_ID} not found"
  }
}
