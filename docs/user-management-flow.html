User Management — Simple Flow

How an admin sets up a new user, and what happens when that user logs in.

1 Admin registers a new user

Everything the admin does to set someone up — from filling in their details to the moment the account is ready to use. 

flowchart TD
  A([Admin starts: add a new user]):::start --> B[Enter personal details:
name, date of birth, gender,
blood group, nationality,
parents' names, ID numbers, address]:::step
  B --> C[Add a profile photo
a small image file · optional]:::step
  C --> D[Enter contact details:
mobile number, email,
emergency contact]:::step
  D --> E[Set login details:
a username and a password]:::step
  E --> F[Choose the user's role
and type: internal or external]:::step
  F --> G[Choose the branches the user works at:
mark one as the main branch,
turn each branch on or off,
pick a default work area for each]:::step
  G --> H[Adjust what the user is allowed to do
in each branch and work area
optional · starts from role defaults]:::step
  H --> I[Admin saves the new user]:::step
  I --> J{Are all the details valid?}:::choice
  J -- No --> K[Show clear messages and let the admin fix them:
· must be at least 18 years old
· ID numbers in the right format
· valid mobile number
· username and email not already used
· only one main branch
· default work area must exist at that branch]:::step
  K --> B
  J -- Yes --> L[Create the account:
· give a unique user number
· keep the password safely scrambled
· keep the ID number protected
· save branches, work areas and allowances]:::step
  L --> M[Record who created the user and when]:::step
  M --> N([Done — the new user can now log in
with their username and password]):::start

  classDef step fill:#e7f5ff,stroke:#1c7ed6,color:#0b3d66;
  classDef choice fill:#fff3bf,stroke:#f08c00,color:#663c00;

2 What happens when the user logs in

The full journey for the newly created user — including every thing that can go wrong and the happy path to their dashboard. 

flowchart TD
  A([User opens the login screen]):::start --> B[Enter login ID and password
login ID can be their mobile,
email, or username]:::step
  B --> C{Does an account with
this login ID exist?}:::choice
  C -- No --> S1[Show: login details are not correct]:::stop
  C -- Yes --> D{Is the account locked for now?
too many wrong tries earlier}:::choice
  D -- Yes --> S2[Show: account is locked,
please try again later]:::stop
  D -- No --> E{Is the password correct?}:::choice
  E -- No --> F[Count this wrong attempt]:::step
  F --> G{Too many wrong attempts now?}:::choice
  G -- Yes --> H[Lock the account for a short while]:::step
  H --> S2
  G -- No --> S1
  E -- Yes --> I{Is the account switched on
for this business?}:::choice
  I -- No --> S3[Show: account is turned off,
please contact the admin]:::stop
  I -- Yes --> J[Reset the wrong-attempt count]:::step
  J --> K[Gather the branches and roles
the user can use
· branches that are switched off are hidden]:::step
  K --> L{Does the user have at least
one switched-on branch or role?}:::choice
  L -- No --> S4[Show: no active access yet,
please contact the admin]:::stop
  L -- Yes --> M{Is a main branch set
for the user?}:::choice
  M -- Yes --> N[Take the user straight into
their main branch and role]:::step
  M -- No --> O[Show the list of available
branches and roles to pick from]:::step
  O --> P[User picks a branch and role]:::step
  N --> Q([Logged in — user lands on the
dashboard for that branch]):::start
  P --> Q
  Q --> R[Any time later, the user can switch
to another of their branches
· only the switched-on ones show up]:::step

  classDef choice fill:#fff3bf,stroke:#f08c00,color:#663c00;
